# 原子团队工作流

使用本 skill 做个人/小团队的软件交付决策。核心目标：

```text
日常轻、边界硬、高风险重、上下文省、自动化单独管。
```

## 决策树

先回答这几个问题，再决定读哪些细节：

1. **这是什么模式？**
   - 新项目：先跑通最小纵向切片。
   - 老项目重构：先保留现有行为，再升级技术或 UI。
   - 功能开发：围绕一个用户能力或一个技术主题。
   - 性能/UI：先有基线，再做同口径对比。
   - 发布/AFK：提高护栏，要求回滚和失败即停。
2. **是否高风险？**
   - 碰到认证、权限、删除、数据迁移、支付/订单/核心业务、上传/导入/导出、异步/队列/流式、跨服务契约、发布、自动化循环，按高风险处理。
3. **是否单主题？**
   - 能清楚证明所有改动服务于同一个用户意图或技术目标，才算单主题。
   - 不能证明时，拆分或询问用户。
4. **是否需要模板？**
   - 只在用户要方案、检查表、契约或发布材料时读取 references；普通建议不要一次性加载全部模板。

## 核心规则

- **单主题**：一个 commit/PR 只表达一个功能、修复、重构、迁移或流程更新。
- **边界硬**：UI、API/业务逻辑、数据/检索/基础设施职责不能混。
- **两档验证**：普通改动跑最小检查；高风险改动跑真实链路或等价 smoke。
- **先基线后重构**：老项目改动前保留截图、接口、性能、运行时证据或已知行为。
- **AI 有边界**：给 AI 的任务要有修改范围、禁止范围、验收标准和验证证据。
- **上下文预算**：先 `rg`，再读小范围；不读长日志、构建产物、数据库或无关大文件。
- **复用优先**：优先更新仓库已有规则、脚本、模板和基线，不创建平行流程。

## 同主题例子

正例：

- `api` 调整接口字段 + `web` 更新调用方 + 文档同步契约。
- 修复 RAG 启动命令 + README 更新启动说明。
- 新增删除确认 API 行为 + 前端接入同一个删除流程。

反例：

- 修聊天滚动时顺手改登录样式。
- 调整 RAG 检索参数时顺手重构 web 组件。
- 发布脚本修复和无关文档改名混在一个 commit。

## 验证选择

普通改动：

- 文档/流程：语法、链接或人工 skim。
- 小 UI：组件/页面 smoke，必要时定点浏览器检查。
- 普通 API：定点请求或相关测试。
- 工具函数：聚焦测试或直接函数断言。

高风险改动：

- 走真实用户/服务链路，或能覆盖同等风险的 smoke。
- 数据迁移需要 dry-run、备份/恢复、幂等或重复执行行为、迁移后校验。
- 发布需要监控、回滚步骤、owner 和观察窗口。

## 模板导航

按需读取，不要默认全部加载：

- 新项目启动方案：`references/新项目启动检查表.md`
- 企业级新项目门禁：`references/企业级新项目门禁.md`
- 老项目重构方案：`references/老项目重构检查表.md`
- 企业级老项目规则：`references/企业级老项目重构规则.md`
- 前后端契约：`references/前后端契约模板.md`
- 发布/回滚：`references/发布回滚检查表.md`

## 不适用场景

- 仓库 harness、agent 入口、指令系统设计：使用 harness 类 skill。
- 多会话里程碑拆解：使用 blueprint 类 skill。
- 纯编码规范、语言风格或具体实现细节：使用更窄的技术 skill。
- 大组织治理、合规体系、PMO 流程设计：使用专门的组织流程方法。

## 输出格式

回答或落地工作流时使用这个顺序：

1. 模式：新项目 / 老项目重构 / 功能 / 性能 / UI / 发布 / AFK。
2. 最小规则：保护当前模式的 3-5 条规则。
3. 高风险项：这个项目的具体风险。
4. 基线：开始前需要的行为、截图、API、数据、性能或非功能证据。
5. 验证：能证明变更的最小检查。
6. 提交/PR 形状：单主题摘要格式。
7. 仓库改动：需要修改的文件、脚本、模板或文档。
8. 剩余风险：仍未完全验证的风险。

如果用户要你改仓库，做最小有用修改，并用轻量检查验证。
